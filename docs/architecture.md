# 系统架构与实施路线

## 当前边界

```text
OrangePi device-agent
  └─ WebSocket：心跳、设备状态、白名单命令
        └─ FastAPI 服务端
             ├─ 设备网关：在线连接保存在内存
             ├─ 业务 API：设备、学生、课程、学习记录
             └─ SQL 数据库：长期状态与业务记录
```

内存只保存当前 WebSocket 连接，数据库保存设备元数据、状态历史、命令结果和学习记录。服务端重启时先将历史在线状态置为离线，OrangePi 的 systemd 服务随后自动重连并恢复在线状态。

## 模块职责

| 模块 | 职责 | 不负责 |
|---|---|---|
| `device` | 采集板端状态、执行白名单命令、断线重连 | 保存云端密钥、执行 Shell |
| `server/app/routes/devices.py` | 设备鉴权、连接管理、状态与命令 API | 教学内容生成 |
| `server/app/routes/learning.py` | 学生、课程、会话、消息和练习记录 | 直接控制硬件 |
| `server/app/models.py` | 持久化数据结构 | 在线连接对象 |
| 后续 `knodo` 适配器 | 封装 Knodo API、超时、重试和响应转换 | 向设备暴露 PAT |

## 部署拓扑

当前局域网联调由 Windows 电脑运行 FastAPI，OrangePi 连接 `ws://192.168.1.18:8000`。电脑关机时服务端不可用，设备代理会持续退避重连。

公网阶段应将 FastAPI 部署到支持常驻 WebSocket 的容器或虚拟机，并使用 PostgreSQL。网页可以独立部署到 Vercel，通过 HTTPS 调用 FastAPI；Vercel 网页部署不替代设备长连接服务。

## 后续顺序

1. 增加 Knodo 服务端适配器和模拟实现，先完成文本对话链路。
2. 将对话输入输出写入现有学习会话和消息表，加入超时、重试与调用追踪。
3. 接入板端麦克风与扬声器，形成唤醒、录音、ASR、智能体、TTS、播放闭环。
4. 接入摄像头和官方 `.nb` NPU 模型，以结构化视觉事件作为智能体上下文。
5. 开发教师管理页和学生学习页，再加入动画、绘本、编程环境与游戏化练习。
6. 最后接入 ESP32 和运动传感器，并实现本地急停、限幅与超时保护。

每一步都应先使用模拟适配器通过端到端测试，再替换为真实外部服务或硬件实现。
