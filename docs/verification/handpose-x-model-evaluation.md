# handpose_x 模型评估

来源：<https://github.com/XIAN-HHappy/handpose_x>

## 结论

这个仓库适合作为“手部关键点回归模型”的候选，不是开箱即用的完整手势识别器。仓库 README 明确说明它不包含手部检测部分；因此完整链路仍需要先检测手部框，再裁剪手部图像，再运行关键点模型，最后由规则或分类器判断张手/握拳。

## 已确认的接口

- 模型输出为 21 个关键点的二维坐标，即 `42` 个数值。
- `model2onnx.py` 的默认导出输入为 `1x3x256x256`，输出名为 `output`，使用 ONNX opset 9。
- ONNX 示例的图像预处理是缩放到 256、`CHW` 排列和 `0..1` 归一化；训练推理脚本另有 `(image - 128) / 256` 的版本，实际转换前必须以权重对应的训练配置为准。
- 关键点模型本身不产生 `open_palm` 或 `fist` 标签，标签需要在 21 点上额外计算。
- 仓库代码采用 MIT License；预训练权重未随仓库提供，而是外部下载地址。

## 与 OrangePi NPU 的差距

当前开发板已经验证 VIPLite/NB 官方样例，但还没有厂商 ONNX-to-NB/Acuity 转换工具、该模型的权重文件、量化参数和输出精度说明。缺少这些材料时，不能声称 `handpose_x` 已经能在 `/dev/vipcore` 上运行。

当前项目使用浏览器本地 MediaPipe WASM 路径完成检测和关键点推理；拿到权重、转换工具和一组授权测试图片后，再增加 NB adapter，并用同一组图片对比 CPU/NPU 关键点误差。

## 需要用户提供的材料

1. 与仓库代码匹配的 `.pth` 权重文件及其来源/许可。
2. 香橙派厂商转换工具版本、命令示例和量化校准格式。
3. 转换后 `.nb` 的输入布局、量化 scale/zero-point、输出张量说明。
4. 不含个人信息的张手、握拳和无手测试图片或视频，用于离线验收。
